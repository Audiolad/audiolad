# Безопасность миграций базы данных

## Правило

Автоматический rollback касается **только приложения**.

База данных автоматически **не откатывается**.

## Требования к миграциям

1. Миграции должны быть обратно совместимыми со старым кодом, пока новый релиз не стабилизирован.
2. Сначала добавляются новые поля и таблицы.
3. Старые поля удаляются только отдельным последующим релизом.
4. Destructive SQL (`DROP`, `TRUNCATE`, массовый `DELETE`) — только с отдельным подтверждением.
5. Перед существенными миграциями создаётся резервная копия.

## Порядок безопасного изменения схемы

1. Добавить новое поле nullable.
2. Задеплоить код, который умеет читать и писать новое поле.
3. Заполнить данные.
4. Сделать поле обязательным отдельной миграцией позже.
5. Удалить старое поле ещё позже отдельным релизом.

## Что делать при проблеме после миграции

1. Откатить **приложение** через `rollback.sh`.
2. Не откатывать базу автоматически.
3. Проверить, совместим ли старый код с новой схемой.
4. При необходимости подготовить forward-fix миграцию.

## В этой задаче

Миграции **не применялись** и **не откатывались**.
