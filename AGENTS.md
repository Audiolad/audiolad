# AGENTS.md — главная инструкция проекта «АудиоЛад»

## 1. Назначение документа

`AGENTS.md` — **обязательная точка входа** в проект «АудиоЛад».

Любой новый исполнитель (ИИ-агент, разработчик, внешний подрядчик) **перед началом любой работы** обязан:

1. прочитать `AGENTS.md` полностью;
2. изучить существующий код и связанные файлы;
3. проверить `git status`;
4. составить краткий план работы;
5. дождаться подтверждения от владельца или архитектора;
6. только после этого вносить изменения в файлы.

Без выполнения этих шагов работа не начинается.

**Стек и версии:** актуальные версии библиотек и инструментов (Next.js, React, Node.js, npm и др.) определяются автоматически по `package.json`, `package-lock.json` и рабочему окружению проекта. Не фиксировать их вручную в этом документе.

**Структура приложения:**

```
/var/www/audiolad/
├── src/app/          # страницы (App Router)
├── src/components/   # общие компоненты
├── src/lib/supabase/ # клиенты Supabase (browser, server, proxy)
├── public/           # статические файлы
├── proxy.ts          # обновление сессии Supabase
```

**Переменные окружения (только имена, без значений):**

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`

Пароли, токены, приватные ключи и значения секретных переменных **не помещать** в этот документ, отчёты и коммиты.

Это **не стандартный Next.js из обучающих материалов**. Актуальная версия в проекте может содержать breaking changes — API, соглашения и структура файлов могут отличаться от устаревших знаний. Перед написанием кода читать документацию в `node_modules/next/dist/docs/` и учитывать deprecation notices.

## Главные принципы проекта

- **Минимальные изменения** — делать только то, что требуется для задачи.
- **Не ломать существующий код** — сохранять работоспособность того, что уже функционирует.

- **Не создавать дубликаты** — не повторять уже существующую логику или файлы.
- **Перед созданием нового файла** — искать существующие решения в проекте.
- **Расширять существующую архитектуру** — вместо создания параллельной.
- **Каждое изменение должно иметь понятную цель** — без цели код не пишется.

## 4. Обязательный процесс работы

Для **каждой задачи** исполнитель следует строгому порядку:

1. **Изучить** существующий код и все связанные файлы.
2. **Проверить** `git status` — зафиксировать незакоммиченные изменения.
3. **Определить** текущее поведение системы (что работает, что нет).
4. **Составить** краткий план: что будет сделано и зачем.
5. **Перечислить** файлы, которые предполагается изменить.
6. **Дождаться подтверждения** от владельца или архитектора.
7. **Внести минимальные изменения** — только то, что нужно для задачи.
8. **Показать** `git diff` по изменённым файлам.
9. **Запустить** доступные проверки (см. раздел 7).
10. **Сообщить результат** — что сделано, что проверено, что осталось.
11. **Обновить документацию**, если изменилась архитектура, база, инфраструктура или состояние проекта.
12. **Зафиксировать точку продолжения** — с какого шага начинать следующую сессию.

## 6. Правила работы с Git

| Правило | Описание |
|---------|----------|
| Основная ветка | `main` |
| Перед работой | обязательно `git status` |
| Незакоммиченные изменения | **нельзя стирать или перезаписывать** чужие правки |
| Перед коммитом | обязательно показать `git diff` |
| Создание коммита | только после подтверждения владельца или архитектора |
| Сообщение коммита | понятное, отражает суть задачи, на русском или английском |
| Запрещено без разрешения | `git reset --hard`, `git clean -fd`, `git push --force` |
| Содержимое коммита | не включать случайные или несвязанные изменения |

Перед каждой рабочей сессией обязательно выполнять `git status` и учитывать фактическое состояние рабочей копии.

## 7. Проверки после изменений

После внесения изменений исполнитель обязан выполнить **минимум**:

| Проверка | Команда / действие |
|----------|-------------------|
| Diff | `git diff` по всем изменённым файлам |
| Линтер | `npm run lint` |
| Сборка | `npm run build` |
| Ручная проверка | пройти изменённый сценарий в браузере |

**Дополнительные правила:**

- Если автотестов нет — прямо сообщить об этом в отчёте.
- Если какая-либо проверка не выполнена или завершилась ошибкой — **не скрывать**, описать причину.
- Перезапуск production через PM2 (`pm2 restart audiolad`) — **только после отдельного подтверждения**.

## 8. Правила работы с базой данных

- Используется **self-hosted Supabase** на сервере Timeweb Cloud (Docker).
- Перед изменением схемы: изучить текущие таблицы, связи, триггеры и политики RLS.
- Актуальная схема базы данных должна храниться в `docs/DATABASE.md` после его создания.
- Любые SQL-команды **сначала показать** владельцу или архитектору.
- Destructive SQL (`DROP`, `TRUNCATE`, массовое `DELETE`) — **запрещён** без отдельного подтверждения.
- Service role key, пароли и секреты — **не выводить** в отчёты, документацию и коммиты.
- Изменения базы должны быть **задокументированы** в `docs/DATABASE.md`.

## 9. Правила работы с инфраструктурой

Перед изменением Nginx, PM2, Docker, портов, домена или SSL:

1. **Изучить** текущую конфигурацию.
2. **Сделать резервную копию** изменяемого файла.
3. **Показать план** изменений владельцу или архитектору.
4. **Получить подтверждение**.
5. **Проверить конфигурацию** до перезапуска (`nginx -t`, `pm2 describe audiolad` и т.д.).
6. **Предусмотреть способ отката** (бэкап, предыдущая версия конфига).

## 12. Текущее известное состояние

### Что работает

- **Регистрация** (`/auth/sign-up`) — форма создаёт пользователя в Supabase Authentication.
- **Профиль в базе** — при регистрации автоматически создаётся запись профиля пользователя.
- **Редирект после регистрации** — после успешной регистрации выполняется переход на `/auth/sign-in?registered=1`.
- **Сообщение после регистрации** — на странице входа при `registered=1` отображается: «Регистрация прошла успешно. Теперь войдите в аккаунт.»
- **Вход** (`/auth/sign-in`) — форма авторизует пользователя, после успеха редирект на `/profile`.
- **Каталог** (`/catalog`) — загружает опубликованные практики из базы через Supabase REST API.
- **Прокси сессий** (`proxy.ts`) — обновляет сессию Supabase на каждый запрос.

### Следующая ближайшая задача

Подключить страницу профиля (`/profile`, `/profile/edit`) к сессии и данным Supabase. Сейчас экран профиля использует демонстрационные данные и не читает авторизованного пользователя.

### Что ещё не связано с реальными данными

- **Профиль** (`/profile`, `/profile/edit`) — отображает статичные демо-данные, не читает сессию Supabase.
- **Большинство экранов** (плейлисты, избранное, история, покупки, кабинет автора и др.) — содержат демонстрационные данные.
- **Навигация к auth-страницам** — ссылки на вход/регистрацию есть только между `/auth/sign-in` и `/auth/sign-up`, в основном UI их нет.
- **Защита маршрутов** — приватные страницы доступны без авторизации.
